01 / FIRST PRINCIPLE

まず、2つのファイルだけ覚える

パスノートにはビルド処理もサーバーもありません。利用者が扱う製品は、画面と処理を持つHTMLと、持ち運べるデータの2つです。

path-note.html

見えるものと、動かすもの

  • 画面の構造
  • 色やレイアウト
  • 追加・検索・編集などの処理
  • ブラウザへの保存処理
＋
data.js

最初のデータと、持ち運ぶデータ

  • 初めて開いたときの内容
  • バックアップ
  • 別のブラウザや人への受け渡し
なぜHTMLを分割しない？
利用者がコピーして使い始めるファイルを、2つのままに保つためです。コードが大きくなる代償は、この説明資料と章コメントで補います。

03 / DATA FLOW

ブラウザで追加したデータは、どこへ行く？

普段の最新データはブラウザ内にあります。data.jsは最初の種と、持ち運び用のファイルです。

data.js初回表示の種。バックアップや受け渡しにも使う。
→
localStorageブラウザで使っている最新データ。追加・編集のたびに更新。
→
新しい data.js残したいときにブラウザから書き出す。
大切な考え方
data.jsを編集しても、すでにlocalStorageへデータがあるブラウザには自動反映されません。普段の正本はlocalStorageです。
04 / TWO DATA SHAPES

なぜ、同じパスを2つの形式で持つ？

保存するときは分類の木構造が便利です。検索や画面表示では、1件ずつ独立したオブジェクトの方が扱いやすくなります。

保存用: 分類の中にあるパス
["共通", [
  ["申請", [
    ["経費", [
      ["経費精算フォルダ",
       "\\\\server\\expense",
       "年度ごとに格納"]
    ]]
  ]]
]]
分類の順番や、まだパスがない空の分類も保持できます。
画面用: 1件ずつ取り出したパス
{
  tab: "共通",
  category: "申請",
  subCategory: "経費",
  name: "経費精算フォルダ",
  path: "\\\\server\\expense",
  note: "年度ごとに格納",
  _meta: { t: 0, c: 0, s: 0, i: 0 }
}
検索、絞り込み、一覧表示が簡単になります。_metaは保存用データ内の元の場所です。

06 / WHY IT IS BUILT THIS WAY

設計判断には、必ず代償がある

「正しい設計」ではなく、パスノートが何を優先したかを記録します。判断を変えるときは、得るものと失うものを再評価します。

JavaScriptとCSSを別ファイルへ分割しない
得られるもの

利用者が扱う製品ファイルを2つに保てる。

代償

path-note.htmlが大きくなり、コードを追いにくい。

対策

責務ごとの章コメントと、この設計ガイドを維持する。

localStorageを日常利用の正本にする
得られるもの

ブラウザ上の追加・編集をすぐ保存できる。

代償

端末やブラウザを越えて自動同期されない。

対策

data.jsの読み込み・書き出しで移行する。

フレームワークを使わない
得られるもの

ビルド不要で、HTMLを開くだけで動く。

代償

状態管理やDOM更新を自分たちで整理する必要がある。

対策

build・apply・renderの役割を分け、変更処理を共通化する。

07 / WORDS YOU WILL SEE

コードを読むための用語集

すべてを暗記する必要はありません。コード検索で見つけた名前の意味を確認する場所です。

catalog

data.jsと同じ、保存・編集用の階層データ。

state.all

検索と画面表示で使う、平坦化されたパス一覧。

catalogModel

階層データへ追加・編集・削除するための操作窓口。

commitCatalogChange

データ変更、保存、失敗時の復元、再描画をまとめる処理。

inspectCatalog

読み込んだデータの形が正しいか確認する処理。

normalize

階層データを画面用のパス一覧へ変換する処理。

build...Html

画面へ表示するHTML文字列を組み立てる処理。

renderAll

タブ、カテゴリ、パス一覧をまとめて描き直す処理。
