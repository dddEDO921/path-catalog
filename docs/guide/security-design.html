SECURITY DESIGN

安全性は、
境界の設計から考える。

詳細設計安全性

パスノートはサーバーを持たないローカルHTMLアプリです。それでも安全性を無視できるわけではありません。ファイル、localStorage、フォーム入力、HTML出力の境界で、何を信用しないかを決めます。

01 / POSITION

この資料の立場

これは注意事項ではなく、境界設計の資料です。
「危ないから気をつける」ではなく、危ない値が入っても実行しない、表示前に変換する、保存前に検査する、破壊的操作だけ止める、という設計判断として扱います。

今の判断

ブラウザでの登録を主導線にし、data.jsはバックアップ兼初期データとして扱います。インポート時はJavaScriptとして実行しません。

完璧ではない点

静的HTMLアプリなので、ユーザーが信頼できないファイルを自分で開くリスクまでは完全に消せません。できるのは、アプリ内の読み込み境界で任意コード実行や不正データ反映を避けることです。

02 / BOUNDARIES

信用境界を分ける

境界入ってくるもの主なリスク守り方
data.js初期データ、インポートファイル任意JavaScript実行、不正形式実行せずJSON部分だけ読む。strict検査する。
localStorage過去に保存したカタログ破損、古い形式、手編集JSON解析と検査を通し、失敗時は初期データへ戻す。
フォーム入力タブ名、カテゴリ名、パス、ノート空データ、HTML混入必須チェックし、表示時にエスケープする。
HTML出力一覧、検索結果、詳細XSS、DOMの意図しない生成動的な値はtoHtmlを通す。
破壊的操作削除、インポート置き換え意図しない消失、復元困難確認を出し、保存失敗時は変更前へ戻す。
03 / DATA.JS

data.jsは実行せず、データとして読む

受け入れる形
window.PATH_CATALOG = [
  ["共通", [
    ["申請", [
      ["経費", [
        ["経費精算", "\\\\server\\expense", ""]
      ]]
    ]]
  ]]
];
ファイルとしてはJavaScriptでも、右辺はJSON互換にします。

設計判断

String.rawのような書きやすい形式は、インポート時の安全な解析が難しくなります。ブラウザ登録を主導線にするなら、書き出し後のdata.jsはJSON互換を優先します。

判断の代償
手書きのしやすさは少し下がります。代わりに、ブラウザで書き出したファイルをそのまま安全に読み戻しやすくなります。
04 / OUTPUT

HTML出力は必ず変換する

危険な境界
const html = `<span>${item.name}</span>`;
値がHTMLとして解釈される可能性があります。
守る境界
const html = `<span>${toHtml(item.name)}</span>`;
利用者由来の値を表示用文字列へ変換します。
テンプレートリテラルは危険ではない。境界を曖昧にすると危険。
HTMLの固定部分と、外から来た値を分けて考えます。値をHTMLへ混ぜる直前が変換ポイントです。
05 / VALIDATION

検査は境界ごとに強さを変える

通常起動は継続を優先

既存データに多少の不備があっても、表示できるものは表示します。警告を出し、利用者の作業を止めすぎないためです。

インポートは安全を優先

外部ファイルで現在のデータを置き換えるため、strict検査にします。不正な形式は一部だけ取り込まず、全体を拒否します。

場面方針理由
起動時警告しながら継続過去データをできるだけ救う。
インポート時strictに拒否外部ファイルで全体を置き換えるため。
フォーム登録時必要項目を必須空データを作らせないため。
06 / USER FLOW

安全性と入力しやすさのバランス

普段は止めない

コピーや保存成功はトーストで軽く伝える。

破壊的操作だけ止める

削除やインポート置き換えでは確認する。

空登録を防ぐ

種類を選ばせてから必要な入力へ進める。

文脈を固定する

追加先が明らかな場合は、タブやカテゴリをロックする。

復元できるようにする

保存失敗時は変更前のcatalogへ戻す。

説明に頼りすぎない

危険な境界はコードで守り、注意書きだけにしない。

07 / LIMITS

今の設計の限界

ファイルを開く行為自体は制御できない

ユーザーが信頼できないHTMLやJavaScriptを直接開くリスクは、パスノート内部だけでは防げません。

localStorageは端末内の保存でしかない

権限管理、監査、同期、バックアップ保証はありません。必要になれば別の保存方式を検討します。

静的HTMLでは自動更新が難しい

セキュリティ修正を利用者環境へ自動配布できません。配布方法の設計が別途必要です。

検査ロジックも保守対象

データ形式が増えれば、通常検査とstrict検査の差分も複雑になります。

08 / REVIEW QUESTIONS

安全性を見直す質問

これは外から来た値か

ファイル、localStorage、フォーム入力を信用していないか。

実行していないか

読み込んだdata.jsをJavaScriptとして動かしていないか。

表示前に変換したか

HTMLへ入れる直前にtoHtmlを通しているか。

置き換え前に検査したか

インポートでstrict検査を通しているか。

失敗時に戻せるか

保存失敗で中途半端なcatalogが残らないか。

確認は必要な場所だけか

安全のために通常操作まで重くしていないか。
