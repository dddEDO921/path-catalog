CSS GUIDE

CSSは、
変数と上書きで読む。

パスノートのCSSはHTML内にあります。ファイル分割はしませんが、章コメント、CSS変数、表示モードごとの上書きで追えるようにしています。

Tailwindは細かい配置を素早く書くために使います。色、テーマ、状態、コンポーネントの意味は生CSS側へ寄せます。
02 / TAILWIND

Tailwind CDNの使い方

パスノートはビルドなしで動かすため、TailwindはCDN版を使っています。HTML内のユーティリティクラスで、レイアウト・余白・グリッド・レスポンシブ指定を素早く書きます。

Tailwindに任せるもの

  • flex / grid
  • padding / margin
  • width / min-width
  • gap
  • position
  • 簡単な文字サイズ

Tailwindに任せないもの

  • テーマ色
  • 表示モード差分
  • コンポーネントの状態
  • カテゴリ色
  • dialogの見た目
  • プロダクト固有の命名
<script src="https://cdn.tailwindcss.com"></script>
<script>
  tailwind.config = {
    theme: {
      fontSize: {
        xxs: "10px",
        xs: "11px",
        sm: "12px",
        base: "13px"
      }
    }
  };
</script>
CDN版の前提
Tailwindを使っていても、ビルド工程はありません。利用者が扱う製品ファイルを増やさないための選択です。
03 / UTILITY CLASSES

Tailwindユーティリティの読み方

種類例意味CSSへ移す目安
レイアウトflex、grid、grid-cols-[260px_1fr]画面の骨格を作る。複数箇所で同じ構造を使うならCSS化する。
余白px-5、py-4、gap-3局所的な間隔を調整する。意味のある部品の標準余白ならCSS化する。
配置sticky、absolute、right-0メニューやヘッダーの位置を決める。状態やテーマと連動するならCSS化する。
表示制御hidden、overflow-y-auto表示・スクロールの基礎。JSが切り替える状態名なら意味のあるクラスを検討する。
レスポンシブsm:grid-cols-2、sm:col-span-2フォームの列数を画面幅で変える。複雑化したらCSSのmedia queryへ移す。

05 / DESIGN TOKENS

まずCSS変数を見る

色や形を部品ごとに直接指定せず、変数を通して指定します。表示モードの差分も、まず変数の上書きで吸収します。

--bg-base画面の土台。
--bg-layerカードやヘッダー。
--accent選択と主要操作。
--border区切り線。
:root,
[data-theme="current"] {
  --bg-base: #0f172a;
  --bg-layer: #111827;
  --accent: #7c3aed;
  --card-radius: 0.75rem;
  --path-bg: rgba(15, 23, 42, 0.35);
}

[data-theme="open"],
[data-theme="compact"] {
  --bg-base: #f4f4f5;
  --bg-layer: #ffffff;
  --accent: #4f477b;
  --card-radius: 0;
  --path-bg: transparent;
}
06 / VARIABLE GROUPS

変数の役割

グループ変数変更時の影響
背景--bg-base、--bg-layer、--bg-sidebar、--bg-control画面全体、ヘッダー、サイドバー、入力欄。
強調--accent、--accent-hover、--accent-soft主要ボタン、選択状態、hover背景。
文字--text-heading、--text-body、--text-muted見出し、本文、補助テキスト。
カード--card-radius、--card-shadowカテゴリカードや一覧のまとまり感。
パス--path-bg、--path-border、--path-radiusパス文字列のチップ表現。
ダイアログ--dialog-backdropモーダル表示時の背景の強さ。
07 / COMPONENT CSS

部品CSSの読み方

TAB

.tab-btn

タブはボタンですが、タブらしく下線で選択を示します。背景を強くしないことで、ヘッダーの情報量を抑えます。

状態: :hover、.active
見る値: border-bottom-color、color
CATEGORY

.cat-btn

カテゴリはサイドバーの行です。左の短いラインを伸ばして、hoverと選択を示します。

状態: .cat-btn:hover .cat-line、.cat-btn.active
色: --cat-color
PATH

.list-row / .path-chip

一覧行は透明な面から始め、hoverで操作可能性を示します。長いパスは折り返せるようにしています。

重要: word-break: break-all
動き: transform: translateY(-1px)
DIALOG

.dialog-shell

フォーム、検索、詳細で共通の面を使います。背景はdialog::backdropで弱めます。

状態: dialog::backdrop
変数: --dialog-backdrop
08 / VIEW MODE OVERRIDES

表示モードは上書きで作る

共通UIを土台にし、表示モード固有の差分だけを下の章で上書きします。

スタンダード

変数定義が基準。暗色、カード、影、チップ表現を使います。

[data-theme="current"]

オープン

カードを消し、見出し線と余白で構成します。個別セレクタでカード周りを上書きします。

[data-theme="open"] .cat-card

コンパクト

カテゴリカードではなく、表形式に近い独自クラスを使います。

.compact-list

[data-theme="open"] .cat-card {
  border: 0;
  background: transparent;
  box-shadow: none !important;
}

.compact-row {
  display: grid;
  grid-template-columns: 78px 110px 220px minmax(0, 1fr) 24px;
}
09 / STATE STYLING

状態の表現

状態使うもの理由
選択中.activeタブ、カテゴリ、表示モードなど、アプリ内状態を明示する。
hover:hoverクリックできることを軽く示す。
focus:focus-visibleキーボード操作時に現在位置を示す。
開閉[aria-expanded]見た目とアクセシビリティ上の状態を一致させる。
無効:disabled押せない理由がある操作を弱く見せる。
.theme-option.active::after {
  content: "選択中";
  color: var(--accent);
}

.subcat-toggle[aria-expanded="false"] .subcat-line {
  width: 6px;
  opacity: 0.35;
}
10 / CASCADE AND IMPORTANT

カスケードと!important

Tailwindのユーティリティクラスと生CSSが同じ要素に当たるため、プロダクト固有の見た目では一部!importantを使っています。乱用するのではなく、Tailwindよりテーマ変数を優先したい場所に限定します。

使ってよい場面避ける場面例
Tailwindの汎用色やborderを、テーマ変数で必ず上書きしたい。単に詳細度で負けたから、とりあえず付ける。.primary-button、.dialog-shell
表示モードの見た目を統一して守りたい。1箇所だけの微調整。.cat-cardの影や角丸。
.primary-button {
  border-color: var(--accent) !important;
  background-color: var(--accent) !important;
  color: var(--text-on-accent) !important;
}

12 / RESPONSIVE CSS

レスポンシブの考え方

560px以下

詳細画面のパス行は縦並びにする。

検索ボタン

狭い画面ではショートカット表記を隠す。

長いパス

word-breakや省略で横はみ出しを防ぐ。

コンパクト表示

列幅が固定なので、狭い画面での確認を必須にする。

ダイアログ

calc(100vw - 2rem)で画面幅に収める。

タッチ操作

ボタンの押せる範囲を削りすぎない。

13 / CHANGE RECIPES

CSS変更の進め方

色を変える

まず変数。

--accentや--bg-baseを変えます。部品CSSを直接変えるのは、変数では意味が合わない場合だけにします。

部品の形を変える

共通UIを見る。

タブなら.tab-btn、カテゴリなら.cat-btn、パスなら.path-chipを見ます。3表示モードへの影響も確認します。

余白だけ変える

Tailwindを見る。

HTML上のpx-*、py-*、gap-*を調整します。ただし同じ調整が複数箇所に広がるなら、意味のあるCSSクラスに移します。

オープンだけ変える

表示モード章で上書き。

[data-theme="open"] ...の形で差分を書きます。共通UIを壊さないようにします。

新しい表示モードを足す

変数、JS、描画をセットで見る。

CSSだけでは足りません。THEME_CONFIG、buildMainHtml、表示モードメニューも更新します。

14 / CSS REVIEW CHECKLIST

確認項目

変数で済むか

直接色指定を増やしていないか。

Tailwindで済むか

単発の余白・配置なのにCSSを増やしていないか。

3モード

スタンダード、オープン、コンパクトで確認したか。

状態

hover、active、focus、disabledを確認したか。

長い文字

長いパスやカテゴリ名で崩れないか。

モバイル

560px以下で横スクロールが出ないか。

意味

見た目の変更がUI/UX判断と矛盾していないか。
