実装処理フロー中級

パスノートはフレームワークを使わないため、起動、描画、イベント、保存、再描画の流れを自分たちで管理しています。まずは関数単体ではなく、処理の順番を押さえます。

基本の循環

1データを読む

2状態を作る

3画面を描く

DOMを正本にしない
画面の見た目は、保存用のcatalogと画面用のstateから毎回作ります。DOMを直接正本にすると、検索・編集・削除で状態がずれやすくなります。

順番関数役割
1loadTheme表示モードをlocalStorageから読む。失敗時は既定値に戻す。

5init選択肢を描画し、イベントを結び、初回の画面を描く。
03 / RENDERING

描画は3段階に分ける

BUILD

build...Html

状態からHTML文字列を作ります。ここではDOMへ触りません。

例: buildTabListHtml、buildMainHtml、buildListRow
APPLY

apply...

作ったHTMLを実際のDOMへ反映します。DOM更新の範囲を小さく保ちます。

例: applyTabList、applyCategoryList、applyMain
RENDER

render...

buildとapplyをまとめます。必要な部分だけ再描画できる入口です。

例: renderTabList、renderCategoryList、renderMain
ALL

renderAll

タブ、カテゴリ、メイン一覧をまとめて描き直します。大きな状態変更後の出口です。

使う場面: 初期化、タブ変更、カテゴリ変更、データ変更後
04 / USER ACTION

操作はイベントから状態変更へ流す

タブをクリックした場合
onTabClick
  → setSelection({ tab, cat: "" })
  → renderAll()
表示対象を変えるだけなので、保存用データは変更しません。
パスを追加した場合
onAddPathSubmit
  → catalogModel.appendItem(...)
  → commitCatalogChange(...)
  → refreshCatalog(...)
  → renderAll()
保存用データが変わるため、必ず共通の保存ルートを通します。
検索した場合
searchInput input
  → getSearchResults(query)
  → buildSearchResultHtml(...)
  → renderSearchResults()
検索結果は一時表示です。catalog自体は変更しません。
表示モードを変えた場合
onThemeAction
  → applyTheme(theme)
  → storage.write(...)
  → renderMain()
同じデータを別の密度で見せるだけなので、メイン一覧だけ再描画します。
05 / DATA CHANGE

データ変更は必ず共通ルートを通す

共通ルートの考え方
const result = commitCatalogChange(() => {
  catalogModel.appendItem(input);
}, { selection: { tab, cat } });
変更処理だけを渡し、保存・復元・再描画は共通化します。

このルートで守ること

  • 変更前のcatalogを複製する
  • 変更後にlocalStorageへ保存する
  • 保存失敗時は変更前へ戻す
  • refreshCatalogで画面用データを作り直す
  • 選択状態を必要以上に変えない
06 / WHEN READING CODE

コードを読む順番

起動を追う

initから読み、初期化の順番を確認する。

入口を探す

クリックやsubmitならbindからイベント名を探す。

状態を見る

catalogを変えるのか、stateだけ変えるのかを分ける。

描画を見る

表示が変わるならbuild、apply、renderの順に追う。

保存を見る

データ変更ならcommitCatalogChangeを通っているか確認する。

失敗時を見る

保存失敗、対象なし、不正データの扱いを確認する。
