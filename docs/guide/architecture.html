コードの詳細へ入る前に、製品ファイル、保存場所、画面用データの関係を理解します。

01 / PRODUCT FILES

利用者へ渡すのは2ファイル

path-note.html

画面、見た目、追加・検索・編集などの処理、ブラウザへの保存処理を持ちます。

data.js

初回表示のデータです。ブラウザから書き出し、バックアップや受け渡しにも使います。

資料は分割する
製品は配りやすさを優先して2ファイルに保ちます。開発資料は、理解しやすさを優先して役割別に分けます。
02 / DATA FLOW

普段の正本はlocalStorage

data.js

→

localStorage

→

新しい data.js

書き出したバックアップ
data.jsを直接編集しても、すでにlocalStorageへデータがあるブラウザには自動反映されません。
03 / TWO DATA SHAPES

保存用と画面用を使い分ける

保存用: catalog
["共通", [
  ["申請", [
    ["経費", [
      ["経費精算", "\\\\server\\expense", "年度ごと"]
    ]]
  ]]
]]
分類の順番と、空の分類を保持できます。
画面用: state.all
{
  tab: "共通",
  category: "申請",
  subCategory: "経費",
  name: "経費精算",
  path: "\\\\server\\expense",
  note: "年度ごと",
  _meta: { t: 0, c: 0, s: 0, i: 0 }
}
検索、絞り込み、一覧表示が簡単になります。

05 / DECISIONS

設計判断と代償

製品のJavaScriptとCSSを別ファイルへ分割しない
優先

利用者が扱う製品を2ファイルに保つ。

代償

HTMLが大きくなる。

対策

章コメントと役割別資料で補う。

フレームワークを使わない
優先

ビルド不要で開くだけで動く。

代償

状態とDOM更新を自分たちで整理する。

対策

build・apply・renderの役割を分ける。
