DESIGN PHILOSOPHY

パスを探す気持ちに、
画面を合わせる。

パスノートは「きれいな管理画面」ではなく、「あれどこだっけ」を早く解決するための道具です。デザイン判断は、整理よりも発見、装飾よりも迷いの少なさを優先します。

01 / PRODUCT FEEL

目指す印象

少し事務的、でも冷たすぎない

ファイルサーバーや社内URLを扱うため、過度にポップにしません。一方で、エンジニア専用ツールのような硬さも避けます。

一覧性と安心感を両立する

パスは長く、量も増えます。余白で読みやすくしながら、必要なときは密度を上げられる選択肢を残します。

分類は主役ではなく道しるべ

カテゴリやサブカテゴリは探す手がかりです。分類の存在感が強すぎると、肝心のパスが見つけにくくなります。

操作は説明されなくても分かる

追加、検索、コピー、詳細表示は何度も使う操作です。見た目だけでなく、置き場所と反応で理解できるようにします。

02 / UX PRIORITIES

優先順位

1探す
検索・分類・視認性
→
2コピーする
クリックで即利用
→
3育てる
ブラウザ上で追加・編集
管理のための管理にしない
タブやカテゴリをきれいに整理することは大事ですが、最優先ではありません。最優先は、必要な場所へすばやくたどり着くことです。
03 / INFORMATION DESIGN

情報の見せ方

タブ

大きな利用文脈。

常に1つが選択されています。カテゴリ追加は選択中タブへ行えばよいため、毎回タブを選ばせません。

カテゴリ

現在地を絞る道具。

サイドバーに置き、画面全体の絞り込みとして扱います。メイン画面では見出しとして再登場し、一覧のまとまりを作ります。

サブカテゴリ

追加操作の文脈。

サブカテゴリの追加や、その中へのパス追加は、対象の近くにボタンを残します。プルダウンで探させると文脈が切れます。

パス行

最終的に使う情報。

表示名、パス、ノート、詳細ボタンの順に情報量を調整します。長いパスは読めるよりも、コピーできることを重視します。

04 / ADDING UX

追加体験の考え方

汎用追加はヘッダー

何もない状態から追加する、タブを増やす、カテゴリを増やす。こうした全体操作は「＋ 追加」に集約します。

文脈追加はその場

サブカテゴリやパスは、今見ているカテゴリ・サブカテゴリへ追加することが多いため、その場のプラスボタンを残します。

空登録させない

まず何を追加するかを選び、その後に必要な入力だけを出します。空の行や空のカードを先に作る方式は避けます。

整理と登録を混ぜすぎない

タブやカテゴリの整理では、必ずしもパス入力まで進みません。管理系の操作とパス登録は分けて扱います。

05 / VIEW MODES

表示モードの役割

スタンダード

スタンダード

最初に触る人向け。カードと影でまとまりを作り、分類構造を把握しやすくします。

オープン

オープン

カード感を減らし、余白と罫線で軽く見せます。視線の負担を減らして読み流せる状態を狙います。

コンパクト

大量データ向け。見た目の豊かさより、同時に見える件数と比較しやすさを優先します。

06 / MICRO INTERACTIONS

小さな反応の意味

HOVER

少しだけ浮かせる

パス行やアイコンはホバーでわずかに移動します。クリックできることを示すためで、派手な演出ではありません。

ACTIVE

選択状態を濃くする

現在のタブやカテゴリは、背景・線・文字の強さで示します。色だけに頼りすぎないようにします。

DIALOG

背景を弱める

ダイアログ表示時は背面を暗く・ぼかして、作業対象を明確にします。

TOAST

完了を軽く伝える

コピーや保存は短い通知で十分です。作業の流れを止めるモーダル通知は避けます。

07 / MOBILE UX

モバイルで守ること

横にはみ出さない

長いパスは折り返しや省略で逃がす。

押せる面積

ボタンは小さくしすぎない。

入力を優先

ダイアログ内では入力欄と保存ボタンを見つけやすくする。

ショートカット非依存

キーボードなしでも検索・追加・コピーできる。

情報量調整

補助表示は必要に応じて隠す。

スクロール前提

1画面に詰め込まず、縦に自然に流す。

デザイン変更時の判断基準

探しやすいか

操作の入口が増えすぎていないか

プラスボタンやメニューが目立ちすぎていないか。

文脈を壊していないか

今見ている場所へ追加・編集できるか。

3モードの役割が保たれるか

単なる色違いになっていないか。

初見で怖くないか

管理画面っぽさが強すぎないか。

大量データに耐えるか
