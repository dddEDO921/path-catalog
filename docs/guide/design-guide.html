DESIGN PHILOSOPHY

パスを探す気持ちに、
画面を合わせる。

パスノートは「きれいな管理画面」ではなく、「あれどこだっけ」を早く解決するための道具です。デザイン判断は、整理よりも発見、装飾よりも迷いの少なさを優先します。

01 / SHOULD BE

こうあるべき姿

記憶の補助であるべき

パスノートは、完璧な台帳を作るための道具ではありません。「たしかこの辺にあった」を支援し、思い出す負荷を下げる道具であるべきです。

探す前に迷わせないべき

検索、カテゴリ、最近見ている文脈のどれから入ってもよい状態にします。利用者に正しい探し方を強制しない設計を優先します。

育てることが負担にならないべき

登録や編集が重いと、すぐ古いツールになります。きれいに分類してから登録するのではなく、必要なタイミングで自然に足せるべきです。

説明なしで使えるべき

社内の非エンジニアが触っても、検索・コピー・追加の入口が分かるべきです。高度な操作はあっても、初回利用の邪魔をしてはいけません。

02 / UX PRIORITIES

優先順位

1見つかる
検索・分類・視認性
→
2すぐ使える
クリックでコピー
→
3続けられる
軽い追加・編集
管理の美しさより、発見の速さ
分類の整合性、見た目の整然さ、登録フォームの厳密さは重要です。ただし、それらは「必要なパスが見つかる」ことより上に置きません。
03 / INFORMATION DESIGN

情報設計の原則

分類は入口であるべき

目的ではない。

タブ、カテゴリ、サブカテゴリはパスへ近づくための手がかりです。分類を目立たせすぎると、利用者は「整理する画面」と誤解します。主役は常にパスです。

検索は逃げ道であるべき

分類を覚えなくてよい。

利用者は必ずしも正しいカテゴリを覚えていません。分類で探せる人にも、キーワードで探す人にも同じツールであるべきです。

一覧は判断できる密度であるべき

広すぎても狭すぎてもだめ。

余白は読みやすさを作りますが、パスが多い場合は視界から情報を押し出します。表示モードで密度を選べることを前提にします。

詳細は二段階であるべき

普段は軽く、必要なら深く。

一覧ではコピーと識別に必要な情報を見せます。編集、削除、長いノートなどは詳細に逃がし、普段の一覧を重くしません。

04 / ADDING UX

追加体験はこうあるべき

最初に対象を選ばせる

空の入力欄を大量に見せるのではなく、まず「パス」「タブ」「カテゴリ」など、追加したいものを選ばせます。

文脈があるなら選ばせない

カテゴリ内のサブカテゴリ追加、サブカテゴリ内のパス追加では、場所はすでに決まっています。改めてプルダウンで選ばせないべきです。

全体操作は一箇所に集める

どこから始めるか分からない状態では、ヘッダーの「＋ 追加」だけを見ればよい状態にします。

整理と登録を分ける

タブを整理したい人に、パス入力まで要求してはいけません。情報設計の操作と、パス登録の操作は分けます。

05 / VIEW MODES

表示モードは役割を持つべき

スタンダード

スタンダード

初見で構造が分かるべきです。カード、影、まとまりを使い、分類の階層を理解しやすくします。

オープン

オープン

読んで探す人のための表示であるべきです。カードの重さを減らし、見出しと線で軽く視線を流します。

コンパクト

大量データを扱うための表示であるべきです。美しさよりも比較、件数、スキャン速度を優先します。

06 / SHOULD NOT

やってはいけないこと

装飾を主役にする

色、影、アイコンがパスより目立つ状態にしない。

追加入口を増やしすぎる

便利そうでも、画面全体がプラスボタンだらけになるなら避ける。

分類を強制する

きれいに分類しないと登録できない、という体験にしない。

色違いだけの表示モードを作る

モードには密度や読み方の違いを持たせる。

説明を前提にする

READMEを読まないと使えない操作は見直す。

管理者目線に寄せすぎる

運用ルールより、日々探す人の体験を優先する。

07 / MICRO INTERACTIONS

小さな反応は控えめであるべき

HOVER

少しだけ浮かせる

パス行やアイコンはホバーでわずかに移動します。クリックできることを示すためで、派手な演出ではありません。

ACTIVE

選択状態を濃くする

現在のタブやカテゴリは、背景・線・文字の強さで示します。色だけに頼りすぎないようにします。

DIALOG

背景を弱める

ダイアログ表示時は背面を暗く・ぼかして、作業対象を明確にします。

TOAST

完了を軽く伝える

コピーや保存は短い通知で十分です。作業の流れを止めるモーダル通知は避けます。

08 / MOBILE UX

モバイルで守ること

横にはみ出さない

長いパスは折り返しや省略で逃がす。

押せる面積

ボタンは小さくしすぎない。

入力を優先

ダイアログ内では入力欄と保存ボタンを見つけやすくする。

ショートカット非依存

キーボードなしでも検索・追加・コピーできる。

情報量調整

補助表示は必要に応じて隠す。

スクロール前提

1画面に詰め込まず、縦に自然に流す。

09 / DECISION CHECKLIST

デザイン変更時の判断基準

探しやすいか

見た目の変更でパス名やパス文字列が埋もれていないか。

操作の入口が増えすぎていないか

プラスボタンやメニューが目立ちすぎていないか。

文脈を壊していないか

今見ている場所へ追加・編集できるか。

3モードの役割が保たれるか

単なる色違いになっていないか。

初見で怖くないか

管理画面っぽさが強すぎないか。

大量データに耐えるか

数十件、数百件でも破綻しないか。
