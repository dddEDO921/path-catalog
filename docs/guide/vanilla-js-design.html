VANILLA JS DESIGN

フレームワークなしでも、
構造は必要。

詳細設計実装

パスノートはReactやVueを使いません。ただし、何も整理しないという意味ではありません。状態、モデル、描画、イベント、DOM更新を分けることで、単一HTMLのまま保守性を保ちます。

01 / POSITION

この資料の立場

Vanilla JSは方針であって、免罪符ではない。
ビルド不要、配布しやすい、HTMLを開くだけで動くという強みを取る代わりに、状態管理、再描画、イベント設計は自分たちで律する必要があります。

今の判断

製品をpath-note.htmlとdata.jsに保つため、アプリ本体はHTML内のJavaScriptで完結させています。

見直し条件

画面状態がさらに増え、変更のたびに副作用確認が難しくなるなら、テスト追加や内部構造の再整理を優先して検討します。

02 / LAYERS

責務を層で分ける

層担当代表守ること
データ正本保存用の階層データcatalog表示都合で直接崩さない。
モデル操作追加・編集・削除catalogModel階層配列への変更を集約する。
画面状態選択中タブ、カテゴリ、平坦化データstateDOMから復元しない。
描画HTML生成とDOM反映build... / apply... / render...生成、反映、再描画を混ぜない。
イベントクリック、submit、キー操作bind、on...Click入口を明確にし、処理を関数へ逃がす。
03 / STATE

状態は少なく、派生は作り直す

考え方
catalog       // 保存用の正本
state.all     // catalogから作る画面用データ
state.tab     // 今見ているタブ
state.cat     // 今見ているカテゴリ
保存したいものと、表示のために作ったものを分けます。

今の弱点

stateは単純ですが、検索、詳細、編集、整理ダイアログなど一時状態が増えると読みづらくなります。今後は一時状態の置き場所をさらに明確にする余地があります。

state.allだけを更新しない
画面だけ変わって保存されない変更が生まれます。保存したい変更はcatalogModelを通してcatalogへ入れます。
04 / MODEL

catalogModelに階層操作を閉じ込める

外から見たい形
commitCatalogChange(() => {
  catalogModel.appendItem({
    tab, category, subCategory, name, path, note
  });
});
UI側は配列の深い構造を知らなくてよい状態を目指します。
避けたい形
catalog[t][1][c][1][s][1].push(item);
階層構造の知識が画面処理へ漏れ、形式変更に弱くなります。
改善余地
現在も_metaや階層配列の知識は一部の処理に見えています。将来データ形式を変えるなら、モデル操作の境界をさらに厚くする必要があります。
05 / RENDERING

描画は build / apply / render で分ける

buildHTML文字列を作る
→
applyDOMへ反映する
→
render必要な再描画をまとめる

得られるもの

HTML生成だけを読める。DOM更新の場所が分かる。表示モードごとの差分を追いやすい。

失うもの

仮想DOMやコンポーネントの差分更新はありません。再描画範囲を自分たちで判断する必要があります。

06 / EVENTS

イベント委譲で再描画に強くする

親に登録する
list.addEventListener("click", (event) => {
  const button = event.target.closest("[data-path-details]");
  if (!button) return;
  openPathDetailsDialog(findItem(button.dataset.pathDetails));
});
一覧を再描画しても、子要素へイベントを付け直す必要がありません。

注意点

data-*属性が増えすぎると、HTMLとイベント処理の契約が見えづらくなります。新しい操作を足すときは、属性名を意味で揃える必要があります。

07 / IIFE

IIFEで小さな閉じた領域を作る

公開するものだけ返す
const menuController = (() => {
  const close = (name) => { ... };
  const toggle = (name) => { ... };
  return { close, toggle };
})();
内部の補助関数や状態を外へ漏らさず、使い方を絞ります。
使いどころ
catalogModel
menuController
noteTooltip
小さな責務があり、内部の細部を隠したい場所に向きます。
08 / WHEN TO RECONSIDER

この設計を見直す条件

状態が増えすぎる

ダイアログや一時編集状態が絡み合い、どこで変わるか追えなくなる。

描画差分が複雑になる

表示モードや条件分岐が増え、build...が肥大化する。

イベント契約が読めない

data-*属性の意味が散らばり、どの操作か分からなくなる。

テストなしでは怖くなる

保存、編集、削除、インポートが絡む変更で手動確認が限界になる。

配布要件が変わる

ビルドやサーバーを許容できるなら、別の設計選択が現実的になる。

複数人で触る

単一HTMLの読みやすさより、責務境界とレビューしやすさを優先する必要が出る。
